# Frame-алгоритмы выбора антенн

Этот документ описывает семейство `Frame-BF`, `Frame-Int` и `Frame-Gen`, а также
самодостаточные варианты `FrameOnly-BF`, `FrameOnly-Int` и `FrameOnly-Gen`.
Все три алгоритма используют одну схему: построить несколько фреймовых стартов,
оценить их точной целевой функцией и улучшить лучшие старты 1-swap локальным
поиском.

Практическое различие такое:

```text
Frame-*     = сильный портфельный вариант: frame-старты + внешние хорошие старты,
              если они полезны для данной цели.

FrameOnly-* = чистый self-contained вариант: только старты, построенные внутри
              frame-схемы, без вызовов solve_h3 / solve_h3_fast.
```

Для `Frame-BF` и `Frame-Gen` эти варианты почти совпадают, потому что основные
старты уже являются фреймовыми. Главное различие находится в interference-цели:
`Frame-Int` использует `S-threshold-Int` и `N-H3-Fast` как дополнительные сильные
инициализации, а `FrameOnly-Int` использует только off-diagonal frame balancing.

## Обозначения задачи

Дана матрица канала

```text
V in C^{N x L}.
```

Строку `V` будем записывать как вектор

```text
a_n = V_{n,:}^* in C^L,
G_n = a_n a_n^*,
G_S = sum_{n in S} G_n = V^* D_S V.
```

Для активного множества `S`, `|S| = K`, коэффициент мощности равен

```text
z(S) = sqrt(P / max_{n in S} ||a_n||_2^2).
```

Эквивалентный канал:

```text
V_eq(S) = z(S) G_S.
```

Тогда

```text
M_S = V_eq(S) V_eq(S)^*
    = z(S)^2 G_S^2.
```

Точные целевые функции:

```text
U_BF(S) = Tr(M_S),

U_I(S) = ||offdiag(M_S)||_F^2,

U_G(S) = det(M_S + sigma I_L).
```

Для численной устойчивости `Frame-Gen` оптимизирует не сам determinant, а

```text
F_G(S) = log det(sigma I_L + z(S)^2 G_S^2).
```

## Фреймовая форма

Если теоретически

```text
V^* V = (1/L) I_L,
```

то можно ввести

```text
U = sqrt(L) V,
u_n = sqrt(L) a_n.
```

Тогда

```text
sum_{n=1}^N u_n u_n^* = I_L,
```

то есть `{u_n}` образует Parseval frame. Для множества `S`

```text
A_S = sum_{n in S} u_n u_n^* = L G_S.
```

Идея frame-подхода:

```text
A_S approx alpha A_all,
alpha = K / N,
A_all = sum_{n=1}^N u_n u_n^*.
```

Если генератор данных немного нарушает идеальную нормировку, сравнивать надо с
`alpha A_all`, а не обязательно с `alpha I_L`.

Для interference это особенно важно. Если

```text
||A_S - alpha I_L||_2 <= eps,
```

то off-diagonal часть автоматически мала:

```text
||offdiag(A_S)||_F^2 <= ||A_S - alpha I_L||_F^2 <= L eps^2.
```

Для `U_G` сбалансированный спектр также полезен: если собственные значения
`A_S` не проваливаются, то determinant не деградирует.

## Почему алгоритм учитывает `z(S)`

Красивые фреймовые surrogate-задачи обычно смотрят только на `A_S` или `G_S`.
В реальной целевой функции есть множитель

```text
z(S)^2 = P / max_{n in S} ||a_n||_2^2.
```

Из-за этого самая мощная строка не всегда полезна: она может увеличить
`max ||a_n||^2`, уменьшить `z(S)` и ухудшить итоговую цель. Поэтому алгоритм
делает `cap-aware` старты: сортирует строки по мощности и пробует окна после
удаления нескольких самых мощных строк.

Пусть

```text
r_n = ||a_n||_2^2,
pi = argsort(r_n) по убыванию.
```

Для offset `o` стартовое окно берется как

```text
P_{o,b} = {pi_o, ..., pi_{o+K+b-1}},
```

где `b` - небольшой buffer. Из этого окна выбирается ровно `K` активных строк.

## Старт 1: top-power / cap-prefix

Самый простой старт:

```text
S = top-K по r_n.
```

Дополнительно проверяются cap-prefix старты:

```text
S_o = {pi_o, ..., pi_{o+K-1}}.
```

Они нужны, чтобы иногда исключить слишком мощные строки и улучшить `z(S)`.

## Старт 2: frame-balanced pool deletion

Берется окно `P = P_{o,b}` размера `K+b`. Нужно удалить `b` строк, чтобы
оставшееся множество было близко к tight subframe.

Эквивалентно выбирается удаляемое множество `T`, `|T| = b`, такое что

```text
A_T approx beta A_P,
beta = b / (K+b),
A_P = sum_{n in P} u_n u_n^*.
```

Тогда

```text
A_{P \ T} = A_P - A_T approx (1-beta) A_P.
```

В коде каждая Hermitian-матрица `G_n` переводится в вещественный feature-вектор:
диагональ плюс действительные и мнимые части off-diagonal элементов. Это
позволяет быстро считать Frobenius-расстояния обычными скалярными
произведениями.

Greedy deletion выбирает очередную удаляемую строку по максимальному уменьшению
остатка

```text
R_t = beta A_P - A_T.
```

После удаления `T` активное множество:

```text
S = P \ T.
```

## Старт 3: D-optimal / log-det surrogate

Опционально используется surrogate

```text
F_lambda(S) = log det(lambda I_L + G_S).
```

Это стандартная D-optimal / mutual-information цель. Она монотонная
submodular, поэтому greedy имеет классическую гарантию `(1 - 1/e)` для этой
surrogate-задачи. Важно: эта гарантия не переносится напрямую на точный

```text
U_G(S) = det(sigma I_L + z(S)^2 G_S^2),
```

потому что есть квадрат `G_S^2` и множитель `z(S)`.

Практически D-optimal старт полезен как robust initialization. В быстром
benchmark-профиле он выключен, потому что на `N=1000, L<=10` frame-balanced и
cap-prefix старты давали то же качество быстрее.

## Старт 4: interference starts

Для сильного портфельного `Frame-Int` дополнительно используются:

```text
S-threshold-Int,
N-H3-Fast.
```

Они дают хорошие начальные множества для малой interference. Затем уже
запускается тот же exact 1-swap refinement по настоящей `U_I`.

Для самодостаточного `FrameOnly-Int` эти внешние старты не вызываются. Вместо них
используются только внутренние frame-старты:

```text
1. offdiag pool deletion:
   выбрать окно P размера K+b и удалить b строк так, чтобы offdiag(G_{P\T})
   был как можно меньше;

2. offdiag greedy add:
   последовательно добавлять строки, минимизируя норму текущей off-diagonal
   суммы;

3. L=2 phase-balanced start:
   равномерно разложить фазы conj(V_{n,1}) V_{n,2} по окружности.
```

После этих стартов `FrameOnly-Int` делает тот же exact 1-swap search по
реальной целевой функции `U_I`. Поэтому это не переименование `H3`, а отдельный
frame-only алгоритм с общей схемой `G_S = sum G_n` и точной локальной
оптимизацией.

## Точная оценка кандидата

Для любого множества `S` алгоритм строит

```text
G_S = sum_{n in S} G_n,
z2 = P / max_{n in S} r_n,
M_S = z2 G_S^2.
```

Далее используется одна из трех точных оценок:

```text
Frame-BF:
score(S) = U_BF(S) = Tr(M_S).

Frame-Int:
score(S) = -U_I(S) = -||offdiag(M_S)||_F^2.

Frame-Gen:
score(S) = log U_G(S)
         = log det(sigma I_L + M_S).
```

Все три алгоритма являются максимизацией `score(S)`.

## 1-swap локальный поиск

После генерации стартов алгоритм сортирует их по точной цели и улучшает только
лучшие `max_refined_starts` стартов.

Один swap имеет вид

```text
S' = (S \ {i}) union {j},
i in S,
j notin S.
```

Матрица обновляется инкрементально:

```text
G_{S'} = G_S - G_i + G_j.
```

На каждом проходе строятся shortlists:

```text
R subset S      - кандидаты на удаление,
A subset S^c    - кандидаты на добавление.
```

После этого все пары `(i,j) in R x A` проверяются точной целевой функцией.
Принимается лучший улучшающий swap. Если улучшений нет, локальный поиск
останавливается.

## Как выбираются shortlists

Для `Frame-Gen` используется приближенный градиент

```text
d/dG log det(sigma I_L + z^2 G^2)
approx 2 z^2 G (sigma I_L + z^2 G^2)^{-1}.
```

По нему считаются row scores

```text
q_n = a_n^* grad a_n.
```

Также используются выравнивающие спектральные признаки: добавить строки,
которые сильнее проецируются на слабый собственный вектор `G_S`, и удалить
строки, которые перегружают сильный собственный вектор.

Для `Frame-BF` используется разложение

```text
Tr((G_S + G_n)^2)
= Tr(G_S^2) + 2 a_n^* G_S a_n + ||a_n||_2^4.
```

Поэтому хороший кандидат на добавление имеет большой

```text
2 a_n^* G_S a_n + r_n^2.
```

Для `Frame-Int` используется feature-вектор off-diagonal части `G_n` и текущий
остаток off-diagonal суммы.

## Связь с arXiv:1107.0088

Работа de Carli Silva, Harvey, Sato рассматривает PSD-задачу:

```text
B_i >= 0,
B = sum_i B_i.
```

Нужно найти веса `y_i >= 0` с малой поддержкой, такие что

```text
B <= sum_i y_i B_i <= (1 + eps) B.
```

Для матриц размера `n x n` и `m` слагаемых их детерминированный алгоритм дает:

```text
support(y) = O(n / eps^2),
time = O(m n^3 / eps^2).
```

В нашей задаче можно положить

```text
B_i = u_i u_i^*,
B = A_all,
m = N,
n = L.
```

Так как `L <= 10`, теоретическая стоимость PSD-sparsification мала:

```text
O(N L^3 / eps^2).
```

Но есть важное ограничение: результат статьи - weighted sparsifier, а в нашей
задаче нужен бинарный вектор

```text
x_n in {0,1},  sum_n x_n = K.
```

Поэтому эту теорию нельзя использовать как финальный ответ напрямую. Правильное
практическое применение:

1. получить веса `y_i` или их приближение;
2. использовать их как importance scores;
3. построить binary-множество размера `K` через rounding / дополнение;
4. запустить exact 1-swap по `U_BF`, `U_I` или `U_G`.

## Оценка сложности текущей реализации

Пусть:

```text
s = число стартов,
p = число улучшенных стартов,
t = max_passes,
r = remove_limit,
a = add_limit.
```

Предвычисления:

```text
row powers: O(N L),
row grams:  O(N L^2),
sort:       O(N log N).
```

Один проход local search:

```text
candidate ordering: O(N L^2 + L^3),
exact swap scoring: O(r a L^3).
```

Итого грубо:

```text
O(N L^2 + N log N + p t (N L^2 + r a L^3)).
```

В benchmark-профиле сейчас обычно:

```text
p = 3,
t = 2,
r = a = 60,
L <= 10.
```

Поэтому `Frame-Gen` получился быстрым: на полном `N=1000, L=1..10` прогоне он
был в среднем быстрее `S-threshold-Gen` и давал больший `U_G`.

## Где можно ускорить дальше

1. **Ввести два профиля: fast и full.**
   Для больших grid-прогонов можно использовать

   ```text
   max_refined_starts = 2,
   max_passes = 1,
   remove_limit = add_limit = 40.
   ```

   Для финального submit/малых случаев оставлять full-профиль.

2. **Полностью векторизовать пары swap.**
   Сейчас код векторизует добавления внутри каждого удаления. При
   `r=a=60` можно сразу собрать все `r*a` пар и оценить их одним batch. Это
   уменьшит Python overhead.

3. **Использовать rank-one формулы для `Frame-BF`.**
   Для BF не нужно строить `G^2` для каждой пары. Можно считать изменение
   `Tr(G^2)` через `a_n^* G a_n`, `r_n^2` и cross-term между удаляемой и
   добавляемой строкой. Это ускорит `Frame-BF`.

4. **Сделать PSD-sparsification initializer.**
   По arXiv:1107.0088 можно строить weighted spectral sparsifier с поддержкой

   ```text
   O(L / eps^2).
   ```

   Например, при `L=10`:

   ```text
   eps = 0.3  ->  O(111) weighted rows,
   eps = 0.2  ->  O(250) weighted rows.
   ```

   Это меньше, чем `K=500` или `K=750`, поэтому такой sparsifier надо
   дополнить до `K` строк через cap-aware greedy. Его можно использовать как
   дополнительный старт и как способ строить shortlists.

5. **Ускорить `Frame-Int` отдельно.**
   `Frame-Int` качественный, но тяжелее из-за дополнительных interference
   стартов. Можно сделать режим `Frame-Int-Fast`: только `S-threshold-Int`
   старт плюс короткий exact swap. Это должно сохранить большую часть
   выигрыша по interference.

6. **Параллелить benchmark cases.**
   Сам алгоритм уже быстрый, но полный grid состоит из независимых `(N,L,K,seed)`
   случаев. Их можно безопасно запускать параллельно по процессам с отдельными
   временными CSV и финальным merge.

## Практический вывод

Для основной цели `U_G` текущий рекомендуемый алгоритм:

```text
Frame-Gen fast profile.
```

Для BF:

```text
Frame-BF или Frame-Gen.
```

На полном `N=1000, L=1..10, 25/50% off` прогоне `Frame-Gen` часто выигрывал
и BF, потому что `U_G` при `sigma=1` сильно коррелирует с BF при сохранении
спектрального баланса.

Для interference:

```text
Frame-Int.
```

Он медленнее, но существенно снижает `U_I`.

Для energy-efficiency:

```text
MISO-EE.
```

Он специально выбирает меньше активных антенн, поэтому выигрывает proxy
`log2(U_G) / active antenna`, но не является лучшим по raw `U_G`.

## Реализованное ускорение без изменения качества

После первичного варианта был ускорен этап точного swap-поиска.

Для `Frame-BF` теперь не строится `G_{S'}^2` для каждой пары `(i,j)`.
Для замены

```text
S' = (S \ {i}) union {j}
```

используется rank-one формула:

```text
Tr(G_{S'}^2)
= Tr(G_S^2)
  + 2(q_j - q_i)
  + r_j^2 + r_i^2
  - 2 |a_i^* a_j|^2,
```

где

```text
q_n = a_n^* G_S a_n,
r_n = ||a_n||_2^2.
```

Затем учитывается реальный power cap:

```text
U_BF(S') = z(S')^2 Tr(G_{S'}^2).
```

Для `Frame-Gen` используется гибридная стратегия:

```text
если L <= 4:
    оценить все пары (i,j) batch-ом
иначе:
    использовать chunk-loop по удаляемым i
```

Это быстрее на малых `L`, но не создает лишний memory overhead на больших `L`.

Контрольная проверка на фиксированных seed показала нулевое изменение
`U_BF`, `U_I`, `U_G` и среднее ускорение:

```text
Frame-BF:  около 1.49x
Frame-Gen: около 1.10x
Frame-Int: около 1.10x
```

Для `Frame-Int` также проверялось уменьшение числа итераций `N-H3-Fast`-старта.
Это ускоряло алгоритм примерно на 6-9%, но ухудшало mean interference примерно
на 5-7%, поэтому такой режим не включен как основной качественный профиль.
